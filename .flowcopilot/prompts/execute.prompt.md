## Stage 1 · 项目研究

在动手编码之前，先充分理解上下文：

1. **阅读设计文档**：使用 `read` 工具读取 `${designDocPath}` 的完整内容，重点关注与本步骤相关的章节（如接口设计、数据模型变更、前端交互设计等）
2. **阅读已有代码**：仔细阅读 `${currentStep.files}` 中已有的代码（如有），理解现有实现逻辑
3. **阅读依赖步骤**：如果本步骤依赖其他步骤的产出，阅读并确认前置产出已就绪
4. **理解项目规范**：确认当前项目的命名风格、目录约定、编码规范

## Stage 2 · 任务规划（Todo List）

在动手编码之前，必须先将本步骤的任务进行细化，并显式生成一份 Todo List（待办清单）：

1. **生成 Todo List**：如果当前环境提供了待办事项管理工具（例如 `manage_todo_list` 等 MCP 工具），请务必调用它来进行规划；如果未提供工具，请在对话回复中以 Markdown `[ ]` 格式输出。
2. **拆解细化**：将本步骤要实现的功能、需要修改的文件等拆解为具体的 3~7 个子任务。
3. **状态追踪**：在后续执行过程中，每完成一项子任务，必须及时将其标记为已完成（`completed` 或 `[x]`）。

## Stage 3 · 代码生成

根据步骤描述和项目研究成果，编写/修改代码：

### 编码规范要求

- ✅ **中文注释**：所有新增代码必须使用中文注释，关键逻辑处必须有注释
- ✅ **风格一致**：与项目现有代码风格保持严格一致（缩进、命名、导入顺序等）
- ✅ **类型安全**：TypeScript 项目中严格类型标注，不使用 `any`（除非已有代码中已有此模式）
- ✅ **最小变更**：只修改本步骤涉及的文件，不改动无关代码
- ✅ **向后兼容**：新增功能不破坏现有功能

| # | 检查项                                           | 状态 | 说明 |
| - | ------------------------------------------------ | ---- | ---- |
| 1 | 是否引入了新依赖？如有，是否已安装并记录？       |      |      |
| 2 | 新代码是否与老数据兼容？（不会破坏现有数据）     |      |      |
| 3 | 是否存在安全隐患？（SQL注入、XSS、敏感信息泄露） |      |      |
| 4 | 是否修改了框架核心 / 基础组件？                  |      |      |
| 5 | 代码质量是否达到商用标准？（非 demo 级别）       |      |      |

| 7 | 是否需要更新已有的配置文件？                     |      |      |

> 如果任何一项为 ❌，**必须先修复再继续**，不得跳过。

## Stage 7 · 生成完成标记文件

**在所有上述阶段全部完成后**，你 **必须** 在最后使用 `edit` 工具创建一个 JSON 标记文件，以表明本步骤已顺利完成。

⚠️ **防乱码与防故障提示**：
禁止且永远不要使用终端命令行工具和重定向（如 `echo text > file.json` 或 `Out-File`）来写入此 JSON 文件！必须且只能通过内置的 `edit` 或 `create_file` 文件调用工具写入，这样能保证所有的中文字段正确地保存为 **标准 UTF-8 编码**，从根源上截断 Windows PowerShell 导致的静默失败和乱码问题。

文件路径：`${completionMarkerPath}`

文件内容（严格遵循此 JSON 格式）：

```json
{
  "stepId": ${currentStep.id},
  "title": "${currentStep.title}",
  "status": "completed",
  "completedAt": "<ISO 8601 时间戳，如 2025-01-15T10:30:00.000Z>",
  "summary": "<简要描述本步骤完成了什么（一句话）>"
}
```

> **重要**：此标记文件是流程引擎判断本步骤已完成的唯一依据。**不生成此文件，流程将无法自动推进到下一步。**

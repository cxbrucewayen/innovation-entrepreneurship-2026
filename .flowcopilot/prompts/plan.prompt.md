# Phase 2 — 技术蓝图与原子化拆解

## 角色设定

你是一位经验丰富的 **技术项目经理 + 高级架构师**。

## 输入

Phase 1 已输出完整的设计文档，保存在以下路径：

> **设计文档路径**: `${designDocPath}`

请先使用 `read` 工具读取该文件的全部内容，充分理解设计方案后再进行拆解。
如果路径为空或文件不存在，请使用 `search` 工具在工作区中搜索 `design.md` 文件。

---

## 任务概述

基于上述设计文档，输出一份精确到原子操作的实施步骤 JSON 文件。要求每步满足"高内聚、低耦合"原则，可独立完成、独立验证、独立回滚。

---

## Step 1 · 现状研判

在拆解之前，先对项目当前状态进行快速研判：

1. **代码搜索**：扫描项目中已有的相关实现（搜索关键类名/函数名/路由），确认设计文档中"可复用"和"需扩展"的判断是否准确
2. **依赖检查**：确认项目当前的依赖版本，排查与新增功能的兼容性
3. **已有测试**：检查现有测试覆盖情况，识别可能需要更新的测试文件

---

## Step 2 · 原子化拆解规则

每个步骤必须满足以下约束：

| 维度   | 要求                                                        |
| ------ | ----------------------------------------------------------- |
| 粒度   | 单个步骤只做一件事（一个文件/一个模块/一个接口）            |
| 内聚   | 步骤内的所有变更属于同一功能域                              |
| 耦合   | 步骤间通过明确的接口/文件依赖关联，不应有隐式的跨步骤依赖   |
| 可验证 | 每步附带验证命令（Curl 命令 / 测试命令 / 类型检查命令）     |
| 可回滚 | 每步附带回滚策略（Git revert 到哪个 commit / 数据回滚 SQL） |
| 有序   | 按依赖关系排列，无依赖的步骤尽量前置                        |

---

## Step 3 · 步骤精排

对拆解后的步骤进行以下优化：

1. **依赖拓扑排序**：确保被依赖的步骤排在前面
2. **风险前置**：高风险步骤（如数据库变更、框架核心修改）尽量前置，便于尽早暴露问题
3. **测试穿插**：在关键模块完成后插入集成测试步骤
4. **里程碑标注**：每 3-5 步标注一个里程碑（可运行/可验证的中间态）

---

## 输出要求

必须分阶段多次写入输出严格遵循以下 JSON Schema 的文件，保存到 `${output.path}`，每阶段写入最多200行内容（太慢容易断开连接），阶段性写入保存的内容文件为temp.json，最后把这个temp.json重命名为我要的文件命名，注意一定要多次调用工具生成temp.json，就是先生成200行，确认文件有了继续再加200行，直到整个json完成再改名：

⚠️ **重要提示（解决Windows下中文乱码与静默失败问题）**：

1. **禁止**使用终端命令行工具（如 PowerShell 的 `echo`、`Out-File` 或其他输出重定向）来生成文件。
2. **必须**使用提供的系统工具（如 `edit`、`create_file` 等 MCP 工具调用）将完整的 JSON 内容以标准 **UTF-8** 编码直接写入目标文件 `${output.path}`。
3. 工具原生支持 UTF-8，直接在 JSON 内容里书写中文即可，切勿画蛇添足地手动进行繁琐转义，防止乱码和文件截断丢失。

```json
{
  "generatedAt": "ISO 时间戳",
  "designDocRef": "${designDocPath}",
  "totalSteps": 0,
  "milestones": [
    { "afterStep": 3, "description": "后端API可用", "verifyCommand": "curl ..." }
  ],
  "steps": [
    {
      "id": 1,
      "title": "步骤标题（动词开头，如'创建XX'/'修改XX'/'配置XX'）",
      "description": "详细描述：做什么、为什么、怎么做、预期结果",
      "category": "backend|frontend|database|config|test",
      "files": ["涉及的文件路径列表"],
      "dependencies": [],
      "verification": {
        "type": "curl|test|typecheck|manual",
        "command": "具体的验证命令",
        "expected": "预期输出/行为描述"
      },
      "rollback": {
        "strategy": "git-revert|sql-rollback|file-restore",
        "detail": "具体的回滚操作描述"
      },
      "risk": "low|medium|high",
      "estimatedMinutes": 5
    }
  ]
}
```

### 质量要求

- 步骤数量不宜过少（不能太粗粒度），也不宜过多（通常 5-20 步为宜）
- 每个步骤的 `description` 必须足够详细，使得一个不熟悉项目的开发者也能独立执行
- `verification.command` 必须是可直接粘贴执行的真实命令
- 不要遗漏任何设计文档中提到的功能点
